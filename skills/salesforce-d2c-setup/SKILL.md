# Salesforce D2C Commerce Setup Playbook

Practical, **DX-first** playbook for standing up a Salesforce **D2C (or B2B) Commerce** store end to end. Drawn from a real build (a coffee-equipment D2C store, "Bean & Brew"); the patterns apply to any catalog. Substitute your own products/brand.

> **Companion reference:** [`reference/commerce-api-and-data-model.md`](reference/commerce-api-and-data-model.md) has the exhaustive catalog — every Connect REST endpoint, the full cart/order/promotions object model, both calc-framework interface tables, the `commerce/*` LWC adapter list, the B2B-vs-D2C matrix, and the payment-framework objects. Load it when you need exact paths/signatures. This file is the *workflow*; that file is the *catalog*.
>
> **Scope note:** this is **Commerce on the core Salesforce platform** (Experience Cloud LWR + the Commerce data model), NOT "B2C Commerce Cloud" (SFRA / PWA Kit / Demandware) — a separate product. The official guide is the *B2B & D2C Commerce Developer Guide*.

## #0 rule: Official-first for COMMERCE PLUMBING; custom LWC for DISTINCTIVE DESIGN
Keep the **standard commerce components for the transactional path** (PDP, cart, **checkout**, search) — never re-implement cart/checkout/pricing/tax; use the official mechanisms (Commerce store, OrderDeliveryMethod, CommercePayments, entitlement policies, Guest Buyer Profile, CartExtension calculators). But the OOTB Commerce LWR **template's marketing pages (home/landing) are a generic placeholder** ("Alpine Energy" banners, footer links all to `/`) — for a professional, on-brand 2026 storefront you SHOULD build custom LWR design components for the homepage/editorial sections. That's not "Frankenstein," it's the supported path (reference §17). Rule of thumb: **custom design components for storytelling/merchandising pages; standard components for the buy-flow.**

## #1 rule: DX/CLI/API first, UI last
Order for every task:
1. **`sf` CLI** — `sf project deploy/retrieve`, `sf data`, `sf apex run`.
2. **Connect/REST API via CLI** — `sf api request rest "/services/data/v62.0/..."` (uses the stored session even though the raw token is hidden).
3. **Metadata XML deploy** (mdapi when source-format type inference fails).
4. **Apex self-callout** (for APIs with no `sf api` path, e.g. multipart uploads).
5. **Browser/Setup UI** — last resort. To drive the org UI headlessly **without a password**, use `sf org open --path "<lightning path>" --url-only` → navigate the browser to that frontdoor URL (auto-authenticates with the CLI token). `sf org display` **redacts** the access token (returns a 54-char placeholder), so raw `curl -H "Authorization: Bearer ..."` 401s — use `sf api request rest` or the Apex session instead.

## #2 rule: pick the calc framework up front (one or the other, never both)
Cart/checkout calculations run on **one** of two mutually-exclusive frameworks — they **cannot be mixed**:
- **Legacy "Checkout Integrations"** — Apex implements `sfdc_checkout.*` interfaces, registered via `RegisteredExternalService` + `StoreIntegratedService`. Fast to mock; what older stores use.
- **Cart Extensions / Cart Calculate** (current **GA**, Salesforce-recommended) — Apex extends `CartExtension.*` classes registered against extension points (e.g. `Commerce_Domain_Tax_CartCalculator`). Required for the Cart Calculate API and finer control.

Default to **legacy** only for a quick mock/demo or a store already on it. Choose **CartExtension** for new builds. Migrating between them is a rebuild of every calculator, not a flag — decide early. (Full interface/extension-point tables + signatures in the reference file §1.) Payment is outside both — always `commercepayments` (§4).

## 1. Store + data model (the spine)
Core objects: `WebStore`, `ProductCatalog`, `ProductCategory`, `ProductCategoryProduct` (product→category junction), `Product2`, `PricebookEntry` (standard + the store's pricebook → `WebStorePricebook`), `BuyerGroup`, `CommerceEntitlementPolicy`, `CommerceEntitlementBuyerGroup` (policy↔group), `WebStoreBuyerGroup` (store↔group), `WebStoreCatalog` (store↔catalog).

- The standard **"Commerce Store (LWR)"** template in some orgs is **B2B-only** (no separate D2C template); the D2C distinction is buyer access + guest, not a different template. Create the store from the standard template either way.
- Products need a `PricebookEntry` in **both** the standard pricebook and the **store pricebook** to be sellable. (8 products × 2 = 16 entries in the reference build.)
- Add products to a category via `ProductCategoryProduct(ProductCategoryId, ProductId)`; rebuild the search index afterward (see §8).
- `Product2` images are NOT a URL field — see §3.

## 2. Catalog distinct from other stores
If one org hosts multiple brands/stores, keep catalogs distinct: separate `Product2` SKUs per brand, add only the brand's products to that store's category, and (if a category was shared) remove the other brand's `ProductCategoryProduct` rows. Use a `Brand__c` picklist on `Order`/`Web_Event__c` as a *dimension* (not a separate identity) when you want one unified customer across brands.

## 3. Product images — the hard part (CMS multipart upload)
`ProductMedia` (the image link) requires `ProductId` + `ElectronicMediaId` (→ **`ManagedContent`**, required) + `ElectronicMediaGroupId`. Gotchas that cost the most time:
- `ManagedContent` is **NOT DML-createable**; `ManagedContentVersion` is **not an SObject**. Images must be created through the **Connect CMS API**.
- The CMS `source.ref` will **NOT** accept a `ContentDocument`/`ContentVersion` Id ("This image reference isn't available").
- The standard media groups already exist per store: query `ElectronicMediaGroup` — you'll find **Tile Image, Product List Image, Product Detail Images** (+ Banner, Attachments). Reuse them; `ElectronicMediaGroup` is not createable.
- CMS workspace = `ManagedContentSpace` (query `SELECT Id, Name FROM ManagedContentSpace`).

**Winning create recipe** (`POST /connect/cms/contents`, multipart):
- JSON part named `content`: `{"contentSpaceOrFolderId":"<spaceId>","title":"...","urlName":"img-...","contentType":"sfdc_cms__image","contentBody":{"sfdc_cms:media":{"source":{"type":"file"}}}}`
  - `source.type` **must be the constant `"file"`**; do **NOT** include `ref` and do **NOT** include `sfdc_cms:altText` (both rejected with `additionalProperties` errors).
- Binary part named **exactly `contentData`** (the API demands this name regardless of `ref`), with `filename` + `Content-Type: image/png`.
- **Order matters**: content part FIRST, then the `contentData` binary part.

**How to send multipart headlessly** (`sf api request rest` can't do multipart; the token is redacted): build the whole multipart body in **Python**, base64 it, and POST from **Apex self-callout**:
- Add a `RemoteSiteSetting` for the org My-Domain (deploy as **mdapi** — source-format `.remoteSiteSetting-meta.xml` throws `TypeInferenceError`; use a `package.xml` + mdapi dir).
- Apex: `req.setEndpoint('https://<mydomain>.my.salesforce.com/services/data/v62.0/connect/cms/contents'); req.setHeader('Authorization','Bearer '+UserInfo.getSessionId()); req.setHeader('Content-Type','multipart/form-data; boundary=...'); req.setBodyAsBlob(EncodingUtil.base64Decode(b64));`
- **Anonymous-apex script cap ≈ 1MB** → upload **one image per run** (a batch of 8 base64 bodies = 207KB and still failed "Script too large" when combined with maps; loop one at a time, ~25KB each).
- Generate images with **Pillow** (available locally) — clean branded PNGs render fine.

**Then link + publish:**
- `ProductMedia` IS DML-createable: `insert new ProductMedia(ProductId=p, ElectronicMediaId=managedContentId, ElectronicMediaGroupId=<group>, SortOrder=n)` — one per group (List/Detail/Tile = 3 per product).
- Publish the content: `POST /connect/cms/contents/publish` body `{"contentIds":[...]}` (field is **`contentIds`**, not `managedContentIds`/`ids`). Returns a `deploymentId` / `#DEPLOYED`.
- Verify per product via `GET /services/data/v62.0/commerce/webstores/{webStoreId}/products/{productId}` → `defaultImage.url` populated + `mediaGroups`. The product **search** API (`POST .../search/product-search`) is what the LIST page reads — confirm it returns `defaultImage` too.
- Direct media URLs require the storefront session; a bare-domain `curl` 301→404s to the wrong site — that's expected, not a defect.

## 4. Payment (mock for demo, real otherwise)
*(Payment uses the `commercepayments` framework regardless of calc framework — request types, objects, client vs server flow in reference §8.)*
- Mock: an Apex `CommercePayments.PaymentGatewayAdapter` that always approves (real adapter branches on `ctx.getPaymentRequestType()`: Tokenize/Authorize/Capture/Refund). `PaymentGatewayProvider` **can't be inserted via Apex DML** — create with `sf data create record`. Then `PaymentGateway` + a `NamedCredential`.
- Wire to the store: `StoreIntegratedService(StoreId, ServiceProviderType='Payment', Integration=<PaymentGateway.Id>)`. Verify: `SELECT ServiceProviderType, Integration FROM StoreIntegratedService WHERE StoreId='...'`.

## 5. Shipping
- `OrderDeliveryMethod` (e.g. "Standard Shipping", `IsActive=true`). The standard checkout can use a flat delivery method without a custom Apex shipping service. For dynamic rates, register a shipping integration like tax (§6).

## 6. Tax (mock = a real store-completeness win)
*(This is the **legacy** `sfdc_checkout` path — see rule #2. On a CartExtension store, extend `CartExtension.TaxCartCalculator` against `Commerce_Domain_Tax_CartCalculator` instead; reference §1.)*
- Apex `class BeanBrewMockTax implements sfdc_checkout.CartTaxCalculations { IntegrationStatus startCartProcessAsync(IntegrationInfo, Id cartId) {...} }` — compute e.g. 8% per `CartItem`, write `CartTax` rows.
- **Gotcha**: `sfdc_checkout.IntegrationStatus` has **no `message` field** — set only `.status` (`SUCCESS`/`FAILED`); log errors via `System.debug`.
- Register: `RegisteredExternalService(DeveloperName, MasterLabel, ExternalServiceProviderType='Tax', ExternalServiceProviderId=<ApexClass.Id>)` then `StoreIntegratedService(StoreId, ServiceProviderType='Tax', Integration=<RegisteredExternalService.Id>)`. Both ARE Apex-DML-creatable. `StoreIntegratedService.ServiceProviderType` enum: Flow, Price, Promotions, Inventory, Shipment, Tax, Payment, Extension.

## 7. Inventory / supply chain
- `ProductInventory`/OCI is a separate (heavy, often UI-gated) enablement. **Without it, products sell as available by default** — fine for demos.
- For Woo-style stock parity, seed `Product2.Stock_On_Hand__c` (this field often already exists from a Woo sync). It's a data layer, not a hard checkout block unless OCI is on.

## 8. Search index
- Rebuild after catalog/category/image/entitlement changes: `POST /services/data/v62.0/commerce/management/webstores/{id}/search/indexes` body `{}`. Poll `GET` the same endpoint until `indexStatus != "InProgress"` (≈2–5 min). The LIST page reads the index; PDP reads live.

## 9. Buyers + entitlement (authenticated)
- A buyer = `Account` → `BuyerAccount(BuyerId, BuyerStatus='Active', IsActive=true)` → `BuyerGroupMember(BuyerGroupId, BuyerId)`. **`BuyerStatus` defaults to `Pending` — must be `Active`** or carts fail "Invalid effective accountId."
- D2C buyers are **Person Accounts**.
- The store's buyer group must have an **active `CommerceEntitlementPolicy`** (`CanViewProduct`/`CanViewPrice`=true) linked via `CommerceEntitlementBuyerGroup`, and the group linked to the store via `WebStoreBuyerGroup`. The note in Setup: "each assigned buyer group must have at least one active entitlement."
- ⚠️ **An entitlement policy with `CanViewProduct=true` but ZERO `CommerceEntitlementProduct` rows entitles NOTHING** — there is no "grant all products" boolean field; the storefront returns **404 on every product** (guest AND buyer) until you insert `CommerceEntitlementProduct(PolicyId, ProductId)` rows for the products (or category rows). This is the #1 "catalog is empty for shoppers but admin API sees products" cause.
- **REST cart as internal admin doesn't work**: `POST /commerce/webstores/{id}/carts?effectiveAccountId=X` returns "Invalid effective accountId" even for a valid Active buyer — D2C checkout runs in the **shopper's storefront session** (guest or community user), not an admin REST context. To prove checkout, log in as a real buyer (recipe below).

### Provisioning a test BUYER LOGIN (to prove checkout headlessly-ish via the live storefront)
A buyer storefront login needs ALL of these — missing any one yields 403/404 on the commerce APIs:
1. **Portal User** on the Person Account (`ContactId = PersonContactId`, profile = the store's CspLitePortal "Shopper Profile"). ⚠️ **The portal account's OWNER must have a `UserRole`** ("portal account owner must have a role") — assign one first. Set the password **in a SEPARATE transaction** from the insert (`System.setPassword` right after insert silently doesn't take for login).
2. **D2C Commerce Shopper permission-set LICENSE** (`PermissionSetLicense` DeveloperName `CommerceShopperPsg`) assigned via `PermissionSetLicenseAssign` — without a commerce PSL the APIs 403.
3. The standard **`CommerceUser`** ("Shopper") permission set (grants the commerce system perm) + a custom permset with **Read** on `ProductCatalog`/`ProductCategory`/`ProductCategoryProduct`/`Product2`/`ProductMedia` (the Customer Community license **rejects `Create WebCart`** — cart objects are handled by the commerce runtime, so don't add cart CRUD).
4. The Account = Active `BuyerAccount` + `BuyerGroupMember`, and the policy has `CommerceEntitlementProduct` rows (above).
5. **Re-login after any permset/PSL change** — community sessions cache permissions; the storefront keeps 403ing on the old session until you log out/in.

## 10. Guest browsing + guest checkout (the saga)
1. **Site public access**: in the site's `DigitalExperienceBundle`, `sfdc_cms__site/<site>/content.json` → `contentBody.authenticationType` must be **`AUTHENTICATED_WITH_PUBLIC_ACCESS_ENABLED`** (valid enum; NOT `AUTHENTICATED_AND_ANONYMOUS`). Deploy (mdapi) + `sf community publish --name "<site>"`.
2. **WebStore flags** (Apex updateable): `OptionsGuestBrowsingEnabled`, `OptionsGuestCartEnabled`, `OptionsGuestCheckoutEnabled`, `OptionsPreserveGuestCartEnabled`; `GuestBuyerProfileId` (object prefix `3K0`, only Id/Name queryable).
3. **CRITICAL — run the official Guest Access automation in Setup UI**: Commerce app → Store → **Settings → Store → Buyer Access tab → Guest Access → Disable → Enable → Continue**. This is the step that **assigns object permissions to the guest profile**, creates the Guest Buyer Profile, sets person-account default, and sets Experience-Builder public access. **Setting the WebStore flags via Apex does NOT run this automation** — the UI banner literally says "Click Disable and then Enable to run the full automation." There is **no headless API** for the guest entitlement/permission assignment.
4. Confirm Store Access → Buyer Groups = your buyer group (re-assigning errors as a duplicate = it's wired).
5. Rebuild the search index; **guest catalog visibility can lag** (entitlement + index recalc; minutes to hours in a Dev Edition org). Even with a complete, correct setup the guest grid may stay empty for a while — verify in an authenticated session meanwhile.
- Drive all the Setup-UI steps headlessly via the `sf org open --url-only` frontdoor (no password).

### Guest visibility = THREE independent gates (all must pass) — the authoritative checklist
A guest 404 on the product API and 403 on pricing almost always means one of these is missing:
1. **Commerce entitlement** — the policy on the store's buyer group needs **`CommerceEntitlementProduct` rows** (no "all products" flag; 0 rows ⇒ 404 for everyone). Guests inherit the **store** buyer group via `WebStoreBuyerGroup`; the **Manage Guest Access automation associates the Guest Buyer Profile with the buyer group** (the guest buyer profile is NOT a `BuyerGroupMember` row — that association is internal, which is why re-running Disable→Enable *after* entitlement products exist is often what finally entitles the guest).
2. **Platform sharing** — modern orgs enforce **"Secure guest user record access"**, which forces the guest to *no* record access regardless of OWD. You need **Guest User Sharing Rules** (Read-only) for the catalog objects. Gotchas learned: `<sharedTo><guestUser>` takes the guest user's **CommunityNickname** (NOT the site name — error otherwise). Share **`Product2`** + **`ProductCatalog`** (Private OWD); `ProductCategory`/`ProductCategoryProduct` are **ControlledByParent** so they follow `ProductCatalog`. **`Pricebook2` rejects guest sharing rules** ("Not available for deploy") — pricebook visibility comes from object **Read** perms + the `ReadSelect` OWD, not a sharing rule. Criteria pattern that matches all: `IsActive equals true` (Product2/Pricebook2) or `Name notEqual zzz_nomatch` (ProductCatalog). **Sharing-rule recalc over existing records is ASYNC** — can lag minutes+; don't conclude failure from an immediate re-test.
3. **Pricebook reachability** — product needs a `PricebookEntry` in the **store** pricebook + policy `CanViewPrice=true`.
- Guest object perms (on the guest profile / a read-only permset assigned to the guest USER): **Read** on Product2/ProductCatalog/ProductCategory/ProductCategoryProduct/ProductMedia/PricebookEntry/WebStore; **Read+Create** on WebCart/CartItem/CartDeliveryGroup/CartTax/Order/OrderItem. ⚠️ You **cannot** assign the standard **`CommerceUser` ("Shopper")** permset to a guest (it has Edit/Update perms; guests allow only Read+Create) — guests get commerce access via the guest profile + the Guest Access automation, NOT that permset.

## 11. Storefront site config (head markup / analytics beacon)
- LWR "Head Markup" (Experience Builder → Settings → Advanced) lives in the bundle at `digitalExperiences/site/<site>/sfdc_cms__appPage/mainAppPage/content.json` → `contentBody.headMarkup`. You can inject a script (e.g. a Data Cloud Web SDK beacon) there, deploy the focused `DigitalExperienceBundle` (mdapi, members `site/<site>` + `content/<space>`), then `sf community publish`. Retrieve with `sf project retrieve start -m DigitalExperienceBundle --target-metadata-dir ...`.
- Publish is async ("you'll get an email"); a row-lock `OracleRowLockedException` means two publishes collided — run ONE `sf community publish` at a time.
- ⚠️ **Data Cloud Web SDK (device-ID + browsing capture) on an LWR storefront — `script-src` is the silent killer, and it's UI-only.** Putting the Data Cloud/Interactions beacon `<script defer src="https://cdn.c360a.salesforce.com/beacon/c360a/<connectorId>/scripts/c360a.min.js">` in head markup is necessary but **not sufficient**: the LWR site's CSP **`script-src`** allowlist (`'self' 'unsafe-eval' 'unsafe-inline'` + a *curated* set: service.force.com, stripe/adyen/paypal, recaptcha, googleapis…) does **not** include the c360a CDN, so the loader is blocked and the console shows **"SalesforceInteractions SDK not loaded"** (the init wrapper runs but the global never appears). To allowlist it you need TWO things:
  1. **connect-src + img-src** for `https://cdn.c360a.salesforce.com` AND `https://*.c360a.salesforce.com` (the SDK *transmits* events there) — **DX-doable** via `CspTrustedSite` metadata (`isApplicableToConnectSrc`/`isApplicableToImgSrc` = true; field order is alphabetical: Connect, Font, Frame, Img, Media, Style — and **`isApplicableToScriptSrc` is NOT a valid element**, deploy errors "invalid at this location").
  2. **script-src** for the c360a CDN (so the loader can *download*) — **NO API PATH EXISTS**. The `CspTrustedSite` *metadata type* has no script-src field, the `CspTrustedSite` *Tooling SObject* exposes only Connect/Frame/Img/Style/Font/Media booleans (no `IsApplicableToScriptSrc`), and the LWR bundle only carries `isRelaxedCSPLevel` (a bool, not an allowlist). Relaxed CSP adds `unsafe-inline`/`unsafe-eval` for *inline* code but does NOT let an external `<script src>` to a non-curated host load. The **only** way to add a third-party host to LWR `script-src` is **Experience Builder → Settings → Security & Privacy → "Trusted Sites for Scripts" (UI-only)**. Verify the live `script-src` directive after: `curl -sI <site>/ | grep -i content-security | tr ';' '\n' | grep script-src` and confirm the host appears. (Contrast: a WordPress/Woo store has no Salesforce CSP, so the identical beacon "just works" there — the LWR CSP is the whole difference.)
- ⚠️ **Button-contrast trap in the OOTB branding sets**: the stock `B2B_Home_Banner` branding set ships with **`ButtonColor: #FFF` on `BackgroundColor: #FFF`** (white button on white bg — only the dark text shows; the button shape is invisible). Fix the brandingSet `contentBody.values` (`ButtonColor`/`ButtonBorderColor`/`ButtonTextColor` + Hover/Focus) in `sfdc_cms__brandingSet/<Set>/content.json`. `B2B_Right_Panel` `TertiaryButtonColor: rgba(0,0,0,0)` (transparent) is *intentional* (subtle tertiary). For custom LWC buttons, avoid transparent "ghost" fills on busy/dark backgrounds — add a translucent fill + a strong (≥0.6 alpha) border so the button reads as a button. ⚠️ **The LWR theme overrides `color`/`background` on `<a>`-based custom buttons** (the storefront ships dark anchor/link colors that win the cascade) — your component CSS renders as dark text on a dark button even though box-shadow/border apply. Fix: use **literal hex colors + `!important`** on `.btn { background/color }` (and on inner `span`s), not CSS custom-property values, so the theme can't override them. Verify via `getComputedStyle` on the live page, not just the screenshot.
- ⚠️ **Embedding MIAW chat on an LWR storefront — the full recipe (4 gates, in order):** raw head-markup alone fails silently. (1) **CSP/inline**: the bootstrap `.js` loads (script-src 'self') but the inline `init()`+`onload` are blocked unless you set the site to **Relaxed CSP** — DX-doable: `sfdc_cms__appPage/mainAppPage/content.json` → `contentBody.isRelaxedCSPLevel=true` (security tradeoff: allows `unsafe-inline` site-wide). (2) **CORS**: the config fetch to `*.my.salesforce-scrt.com` is CORS-blocked → add the site origin to the org **CORS allowlist** (`CorsWhitelistOrigin` metadata, `urlPattern=https://<domain>.my.site.com`). (3) **Init timing**: `init()` on the script's `onload` runs before the LWR app container exists → "Error creating Agentforce Messaging frame: …appendChild of null". Fix: **defer** — inject the bootstrap into `document.body` on `window load` + a ~1s `setTimeout`, then `init()` after another ~1s. Frame then creates. (4) **frame-ancestors / clickjack**: the chat iframe (served from the ESW Experience site) has a `frame-ancestors` allowlist; if the embedding storefront's origin isn't listed, the launcher iframe is blocked from rendering ("Framing '…my.site.com/' violates frame-ancestors"). **THIS IS DX-DEPLOYABLE (don't waste time in the UI):** the allowlist is the ESW site's **`CustomSite.siteIframeWhiteListUrls`** field (each `<siteIframeWhiteListUrls><url>host</url></siteIframeWhiteListUrls>` becomes one frame-ancestors token — host only, no scheme; wildcards like `*.foo.com` allowed). Retrieve `CustomSite:ESW_<deployment>_<numbers>`, **append** your storefront host (e.g. `orgfarm-xxxx.develop.my.site.com`) keeping the existing entries, deploy — the live `frame-ancestors` header updates immediately (verify: `curl -sIL <eswSiteUrl> | grep content-security | tr ';' '\n' | grep frame-ancestors`). Note: `EmbeddedServiceConfig` itself has NO domain field (only `AreGuestUsersAllowed`) and the org Session-Settings "Trusted Domains for Inline Frames" is a *different* (VF/Survey/Lightning-Out) list — the per-deployment allowlist lives ONLY on the ESW `CustomSite`. `clickjackProtectionLevel=SameOriginOnly` does NOT auto-allow a same-host Experience site (the header lists the whitelist explicitly, no `'self'`), so you must add the host even when storefront + ESW share `my.site.com`. The LWR commerce layout also has native messaging hooks (`setupEmbeddedMessagingLocale`) that integrate once the bootstrap is ready. ⚠️ **Brand caveat:** the chat's MessagingSession channel = the deployment's channel, so a SHARED deployment stamps ITS brand — embedding store-A's deployment on store B makes the agent recommend store-A's products. For correct per-store recs give each storefront its OWN messaging channel/deployment mapped to that brand (see the brand-stamp trigger pattern).

## 12. Promotions + coupons (a best-in-class store has at least one)
The platform has a built-in promotion engine — you usually do **not** write a custom promotions calculator. Objects (all Apex-DML-createable; full model in reference §5):
- `Promotion` (the discount; product/shipping/order type) → `PromotionTarget` (what's discounted, the "get Y") + `PromotionQualifier` (conditions, the "buy X") (+ `PromotionTier` for spend/qty thresholds).
- **Automatic** promo = no coupon, applies when qualifiers met. **Manual** promo = attach a `Coupon` (shopper enters a code).
- Scope it to your store/segment via `PromotionSegment` → `PromotionSegmentSalesStore` (store) and/or `PromotionSegmentBuyerGroup` (group). Without an active segment row linking the store, the promo won't surface.
- Applies at the cart via **Evaluate Cart** (`POST /commerce/promotions/actions/evaluate`) on every cart change + checkout; adjustments materialize as `CartItemPriceAdjustment` (line) / `WebCartAdjustmentGroup` (cart) and carry to `OrderItemAdjustmentLineSummary` post-order.
- ⚠️ **`Coupon.Status` defaults to `Inactive`** on DML insert → coupon shows "invalid" at cart until you set `Status='Active'` (+ give it `StartDateTime`/`EndDateTime`).
- ⚠️ A `PromotionSegment` links to a store **OR** a buyer group, not both ("associate to either … but not both"). Authenticated buyers resolve via the **buyer group** — so for a promo to reach logged-in buyers, create a buyer-group-scoped segment (`PromotionSegmentBuyerGroup`) in addition to the store one, and link the Promotion to each via `PromotionMarketSegment`. `PromotionQualifier.MinimumAmount` must be **> 0**.
- ⚠️ **`Promotion.IsAutomatic` defaults `false`** on DML insert → an auto-apply (no-coupon) promo never fires until you set `IsAutomatic=true`. Visible in the Commerce **Promotion Workspace** as an "Automatic = False" column.
- `PromotionTier` is **only for tiered promos** — adding one to a normal promo errors *"Select a tiered promotion"*. A single-discount promo correctly has its `PromotionQualifier` + `PromotionTarget` directly on the `Promotion` (no tier). So a missing tier is NOT the cause of a non-applying simple promo.
- ⚠️ **The real promotions trap = the CartExtension cutover silently drops the promotions calculator.** Diagnosed end-to-end: the DML promo records were 100% correct AND fully recognized by the Commerce **Promotion Workspace** (Discounts tab showed the $150 Fixed-Amount-Off target, Qualifiers/Segments populated, Active+Automatic) — so it is NOT a "DML doesn't compile" problem. Yet a forced cart recalc (qty change → **tax recalculated to exactly 8%, proving the orchestrator runs**) produced **zero discount**, and the coupon stayed "invalid," even after toggling IsActive off/on + reindexing. Root cause: when you enable Cart Calculate (`WebStore.OptionsCartCalculateEnabled=true`) and register only a **Tax** `CartExtension` calculator (+ remove the legacy framework), **the promotions domain has no calculator running** — pricing + tax run, promotions don't. **Fix path**: register/author a **`Commerce_Domain_Promotions_CartCalculator`** (CartExtension) for the store, or keep promotions on the legacy framework (the two are mutually exclusive — so migrating *one* domain to CartExtension can break promotions). Decide tax-framework vs promotions up front; don't partially cut over a live store. (Verify via a live cart — tax recalculating but no promo adjustment is the tell.)
- ⚠️ **Reverting an LWR store from CartExtension → legacy breaks checkout** if it has no legacy `sfdc_checkout` Price/Shipping/Inventory integrations: setting `OptionsCartCalculateEnabled=false` made checkout throw **"integration error in COMPUTE_SHIPPING / can't deliver to this address"** (CartExtension had been providing those calculators as defaults). On a CartExtension-native LWR store, **stay on CartExtension** and add the missing domain calculators rather than reverting.
- ⚠️ **A custom `CartExtension.PromotionsCartCalculator` that adds a *line-level* `CartItemPriceAdjustment` may run cleanly (confirm via a TraceFlag on the buyer user → cart-calc logs as `UniversalPerfLogger`) yet NOT reduce the cart's displayed Subtotal/Total** — a transaction/order-level discount likely needs a **cart-level adjustment (`CartAdjustmentGroup`)** and/or the cart-level total fields set, not just `cartItem.setTotalPriceAfterAllAdjustments()`. Budget iteration for the adjustment-rollup mechanics; "calculator runs + creates the adjustment but the grand total is unchanged" is the tell.
- **Verify live**: evaluation runs in the shopper session, so a promo only proves out in the live storefront cart. Coupon redemption counts move via `…/increase-use/coupon-codes` / `…/decrease-use/coupon-codes`.

## 13. Order Management / OrderSummary (post-purchase — don't skip)
Placing an order (`POST /checkouts/{id}/orders`) creates an **`Order`** (+ `OrderItem`, `OrderDeliveryGroup`). **Salesforce Order Management** then must create the **`OrderSummary`** (auto-generating `OrderItemSummary` + `OrderDeliveryGroupSummary`). The storefront **order-history, reorder, and returns** APIs (`/commerce/webstores/{id}/order-summaries/…`, `commerce/orderApi`) read **OrderSummary, not Order**.
- **Diagnostic**: `SELECT COUNT() FROM Order` ≫ `SELECT COUNT() FROM OrderSummary` means OM isn't summarizing orders → the storefront "My Orders"/reorder is empty even though orders exist. This is the single most common "store looks done but post-purchase is broken" gap.
- Fix: enable **Order Management** features + the **"Create Order Summary"** flow (the standard `Process Order Summary` / OM connect action). The cart→order place step can be configured (CartExtension `Commerce_Domain_Checkout_CreateOrder`) but OrderSummary creation is an OM capability, partly Setup/feature-gated.
- Reorder is first-class once summaries exist: `POST /order-summaries/{id}/actions/add-order-to-cart` (or `startReOrder` in `commerce/orderApi`). Guest order lookup = `/order-summaries/actions/lookup` (email + phone).

## 14. Custom storefront components (LWC on LWR)
Stay with standard components (rule #0); when you genuinely need custom, build LWC against the `commerce/*` modules — never re-fetch via raw Apex/REST when a wire adapter exists. Key modules (full list in reference §6): `commerce/cartApi` (`CartSummaryAdapter`, `addItemToCart`, `applyCouponToCart`…), `commerce/productApi`, `commerce/contextApi` (`SessionContextAdapter` → webstoreId, guest-vs-auth, currency), `commerce/checkoutApi`, `commerce/orderApi`.
- Deploy via SFDX (`sf project deploy start`). `.js-meta.xml`: `<isExposed>true</isExposed>` + `<targets>` of `lightningCommunity__Page`/`_Page_Layout`/`_Theme_Layout`; add `lightningCommunity__Default` + `<targetConfigs>` for editable props in Experience Builder.
- **LWR disables Lightning Locker + Lightning Web Security** — opt into relaxed CSP with `lightningCommunity__RelaxedCSP` under `<capabilities>`.
- Components render repeatedly as the data tree fills — guard for undefined data and re-render; don't assume first render has cart/product data.
- Checkout step customization: child components implement the **`useCheckoutComponent` mixin** (form validation + external-API sync). Cart/checkout-mutating calls must run **sequentially** — concurrent WebCart writes throw **Version Mismatch / Checkout Conflict**.
- Clone-and-customize base: `forcedotcom/commerce-on-lightning-components` (Public Commerce LWR Library).

- Authenticated buyer: products + images + price + cart + checkout (payment+shipping+tax lines) → Order. **Proven recipe**: log in as a provisioned buyer (§9), add a product, checkout → enter a ship-to address → the **tax line appears** when you "Continue to Shipping Method" (that's when cart-calc/tax runs). ## 17. Best-in-class custom storefront DESIGN (homepage / editorial, LWR)
The OOTB template homepage is placeholder filler. To ship a real branded 2026 storefront, build custom LWR page components and swap them onto the home view. Proven, DX-first workflow (no Experience-Builder drag-drop needed):
1. **Build the components** under `force-app/main/default/lwc/<name>/` — `.js` (props via `@api` with sensible defaults), `.html`, scoped `.css` (own design tokens at `:host`; LWR scoped CSS doesn't share, so repeat the palette per component), and `.js-meta.xml` with `<isExposed>true</isExposed>` + targets `lightningCommunity__Page` + `lightningCommunity__Default` + `<targetConfigs>` for editable props. Reference build: a coffee-equipment set — `bbHero` (full-bleed gradient hero + trust stats), `bbValueProps`, `bbFeaturedCollection` (cards linking to **real PDPs** `/store/product/x/<productId>`), `bbBrandStory` (editorial split), `bbNewsletter`. Premium look = CSS gradients/animation (no external images needed), a serif display + sans pairing, generous whitespace, rounded cards, hover lifts.
2. **Deploy** the bundles (`sf project deploy start -m "LightningComponentBundle:bbHero" ...`).
3. **Place them on the home page via metadata** (more reliable than the Builder canvas): `sf project retrieve start -m "DigitalExperienceBundle:site/<Site>" --target-metadata-dir <dir>`, unzip, edit `digitalExperiences/site/<Site>/sfdc_cms__view/home/content.json`. The tree is nodes of `{definition, id, type, attributes, children}`. A full-width section = `community_layout:section` whose **`id` MUST equal `attributes.sectionConfig.UUID`**, with a child `region` (the column, `columnWidth:"12"`) whose **`id` equals the column UUID** in sectionConfig; put your component `{ "definition":"c:bbHero", "id":"<uuid>", "type":"component", "attributes":{"dxpStyle":{}} }` in the column's `children`. Replace the main region's children with one section per component (generate fresh UUIDs).
4. **CRITICAL variant gotcha**: the home view has `mobile/mobile.json` + `tablet/tablet.json` holding per-component attribute OVERRIDES keyed by the OLD component IDs. After swapping the desktop tree, deploy fails with *"component <id> can't be included in the translation variant…"* — fix by setting `contentBody.component.children = []` in both variant files (CSS-responsive components need no overrides).
5. **Premium fonts via head markup** (`sfdc_cms__appPage/mainAppPage/content.json` → `contentBody.headMarkup`): prepend a single `<link rel="stylesheet" href="https://fonts.googleapis.com/css2?family=...">`. The LWR head-markup validator **rejects `<link rel="preconnect">` and `crossorigin`** — use only the plain stylesheet link, and keep solid system-font fallbacks (Georgia serif / system sans) in case CSP blocks it.
6. **Deploy the edited bundle** (`sf project deploy start --metadata-dir <dir>`) then **`sf community publish --name "<Site>"`** (async — "you'll get an email"; wait ~45s before viewing). Verify by loading the storefront home.

### Replacing the stock template CHROME (header/footer/logo) with a custom theme layout — fully DX
The OOTB storefront header/footer + the **logo are template** — the `b2b-storefront` template's **`commerceLayout`** theme layout (built from `commerce_builder:layoutHeaderOne` + `dxp_content_layout:siteLogo` + `experience:megaMenuNavigation` + `multiCartBadge` + footer `linkList`/`socialLinks`), and the logo is the bundled **`/assets/images/logo-alpine-group.svg`** ("Alpine Group" demo brand, set in the `B2B_Commerce` brandingSet `SiteLogo`/`_SiteLogoUrl`). To go fully custom (no template chrome, custom logo):
1. **Build a custom theme-layout LWC**: `isExposed=true`, `<targets>lightningCommunity__Theme_Layout</targets>` (NO `<targetConfig>`/`<property>` — unsupported for this target → deploy error), a default **`<slot></slot>`** for the page content, your own `<header>`/`<footer>` around it, an **inline SVG logo** (sidesteps the asset entirely), and commerce wiring via static imports `commerce/cartApi` (`CartSummaryAdapter` → cart badge) + `commerce/contextApi` (`SessionContextAdapter` → guest-vs-auth account link). Use literal colors + `!important` (theme override, see gotchas).
2. **Create a theme-layout INSTANCE** that points to it: `digitalExperiences/site/<Site>/sfdc_cms__themeLayout/<name>/` with `_meta.json` (`apiName`, `type:sfdc_cms__themeLayout`, `path:themeLayouts`) + `content.json` = `{type, title (no '&'), urlName (hyphenated, no spaces), contentBody:{component:{definition:"c:bbThemeLayout", id:<uuid>, type:"component", attributes:{}}}}` (mirror the simple `externalLayout` instance, not the deeply-nested `commerceLayout`).
3. **Remap the theme**: in `sfdc_cms__theme/<Theme>/content.json`, change the `layouts[]` entry whose `layoutType` is **`Inner`** (the storefront shopping pages: home/PDP/cart/category/search) from `layoutId:"commerceLayout"` → your instance. (`Checkout`/`MyAccount`/`External` types have their own layouts — switch those too only if you rebuild those flows.)
4. **Deploy bundle + publish.** Verify the custom header/footer render and the page content shows in the slot (pierce shadow DOM: `.bb-header`, `.bb-logo__word`; confirm `img[src*="alpine"]` is gone). The page content renders inside your `<slot>`; if pages go blank, the slot is missing/misnamed.

## 18. Brand-aware chat Concierge — scope recommendations to the SITE (multi-brand)
When one org runs **two storefronts** sharing **one** Agentforce/MIAW Concierge (e.g. a Woo coffee site + this D2C equipment site), the chat must recommend from the **catalog of the site the shopper is on** — *without* fragmenting the customer: **brand is a session signal that scopes the catalog; identity stays unified** (personalize from the cross-brand profile). Pattern proven end-to-end:

**A. Make the reco engine brand-aware (Apex).** Add an optional `brand` to your recommendation invocable (`@InvocableVariable String brand`); normalize it and branch the candidate query on `Product2.Brand__c` (the same picklist used as the Order/Web_Event dimension, §2). Keep the existing path as the default so a blank brand still works (e.g. coffee). New-field gotcha: a freshly-deployed `Brand__c` is **not** SOQL/Apex-readable until FLS is granted via a permission set — grant it before backfilling/querying.

**B. Wire the agent to read brand from the session (Agent Script authoring bundle).** Add a **linked** variable sourced from a MessagingSession field, then pass it into the recommendation action on EVERY call:
```
brand: linked string
    description: "The SITE/store this chat is on. Scopes recommendations. Never name the brand to the shopper."   # ≤255 chars or publish fails "data value too large"
    source: @MessagingSession.Brand__c
```
Add `brand: string` to the reco action's `inputs:` block and `brand=@variables.brand` to the action-call instruction. Then `sf agent publish authoring-bundle --api-name <Bundle>` **and** `sf agent activate --api-name <Bundle> --version N` — **publish ≠ activate**; publish leaves the new BotVersion *Inactive* while the old one keeps serving. Verify the active version: `SELECT VersionNumber,Status FROM BotVersion WHERE BotDefinition.DeveloperName='<Bundle>' AND Status='Active'`.

**C. Get brand onto the session — TWO transports. Prefer the trigger.**

**C1 (RECOMMENDED — robust, 100% CLI, no UI publish): derive brand from the CHANNEL** (the site *is* the channel). MessagingSession exposes `MessagingChannel.DeveloperName`. A **before-insert** trigger stamps `Brand__c` at session creation — early enough for the linked var (which resolves at **session START**):
```apex
trigger MessagingSessionBrandStampTrigger on MessagingSession (before insert) {
    MessagingSessionBrandStamp.stamp(Trigger.new);   // handler: map channel DeveloperName -> brand, set Brand__c when blank
}
```
Handler keeps a `Map<String,String>` of `MessagingChannel.DeveloperName -> brand`; query `MessagingChannel` by the sessions' `MessagingChannelId`, set `Brand__c`. Wrap in try/catch that never rethrows (a before-insert trigger that throws blocks ALL chat). **Adding a new storefront's chat = one map line.** If a routing flow also writes `Brand__c`, *remove that assignment* so it can't clobber the trigger with a blank. Cover the handler logic with a pure `brandFor(devName)` unit test; the 1-line trigger body is hard to cover (MessagingSession is runtime-created, near-impossible to insert in Apex) — Developer-Edition `NoTestRun` deploys don't enforce per-trigger coverage, so deploy without `--test-level`.

**C2 (per-session prechat param — needed only if ONE channel serves BOTH brands; PUBLISH-GATED):** pass brand from the embed snippet as a hidden prechat field. Wiring: MessagingChannel `customParameters` (`externalParameterName` = the routingAttributes key the client sends; `actionParameterName` = the routing-flow var) + a routing-flow `Update Records` writing `Brand__c` + an `EmbeddedServiceConfig` `embeddedServiceFormFields` entry (`formField` = the channel param `<name>`, `isHidden=true`, `messagingChannelParameterType=Custom`). **CRITICAL:** all that metadata deploys clean but the live SCRT runtime **silently drops a NEW param until the Embedded Service Deployment is *Published*** (Setup-UI-only; no `sf`/metadata trigger). The runtime reads a *published snapshot*, not the live channel metadata. So C2 always needs one UI publish — which is why C1 (the trigger) is preferred. (Confirm landing with a sentinel probe: create a conversation with `routingAttributes={"<externalParameterName>":"ZZTEST"}` then `SELECT Brand__c FROM MessagingSession ORDER BY CreatedDate DESC LIMIT 1`.)

**MIAW gotchas that cost time here:** (1) there can be **multiple channels** (e.g. `*_Web_Chat` and `*_Web_Chat_V2`) — the LIVE one is the SCRT `developerName`/ESDEV your snippet uses, not necessarily the obvious name; edit that one. (2) **Linked vars resolve at session START** — setting the field mid-session (after create) is NOT picked up; it must be present at insert (trigger / published param). (3) The **agent's running user needs FLS read** on the MessagingSession field — add it to the same permset(s) that already grant the working identity fields (find them: `SELECT Parent.Label FROM FieldPermissions WHERE Field='MessagingSession.<existing identity field>'`). (4) `sf data update record --values` splits on spaces — quote values with spaces/`&`: `--values "Brand__c='Bean & Brew'"`.

Headless live proof (no storefront UI, no AI-Evaluation storage burn): drive the SCRT2 REST + SSE API directly (`/iamessage/v1/authorization/unauthenticated/accessToken` → `/conversation` with `routingAttributes` → `/conversation/<id>/message` → poll `/eventrouter/v1/sse`); assert the agent returns a product from the EXPECTED brand's catalog (e.g. an equipment SKU for the D2C site, not coffee).

## Gotcha quick list
- `sf org display` redacts the token → use `sf api request rest` / Apex session.
- `ManagedContent` not DML-able; CMS image = multipart `contentData` part, `source.type:"file"` no `ref`, no `altText`, content part first.
- Anonymous apex ≈1MB cap → one image upload per run.
- `RemoteSiteSetting` source-format fails type inference → deploy mdapi.
- `IntegrationStatus` has no `message` field.
- `BuyerAccount.BuyerStatus` defaults Pending → set Active.
- `authenticationType` enum = `AUTHENTICATED` | `UNAUTHENTICATED` | `AUTHENTICATED_WITH_PUBLIC_ACCESS_ENABLED`.
- Guest object-permission assignment is the **Disable→Enable Guest Access automation** (UI-only); Apex flags don't trigger it.
- REST cart with `effectiveAccountId` fails for internal-admin context; checkout needs the shopper session.
- `sf org open --url-only` = headless org-UI auth with no password.
- The two calc frameworks (`sfdc_checkout.*` vs `CartExtension.*`) are **mutually exclusive** — don't register both; the Cart Calculate API is incompatible with legacy integrations.
- `PaymentGatewayProvider` is **not** Apex-DML-insertable → `sf data create record`. `PaymentGateway` needs `MerchantCredentialId` → a NamedCredential. Debug via `PaymentGatewayLog`.
- `CartDeliveryGroup.DeliveryMethodId` **deprecated v64, removed v66** — select shipping via `CartDeliveryGroupMethod`.
- **`Order` count ≫ `OrderSummary` count** = Order Management isn't summarizing → storefront order-history/reorder/returns are empty. Enable OM + the Create Order Summary flow.
- A catalogued product not in the **store pricebook** isn't sellable; a category under a catalog **not linked via `WebStoreCatalog`** never shows. Audit both when products are "missing."
- Promotions need an active `PromotionSegmentSalesStore`/`…BuyerGroup` segment row, not just a `Promotion` record, to surface on the store.
- LWR templates disable Locker + LWS → custom LWC needs `lightningCommunity__RelaxedCSP` capability; concurrent cart writes throw Version Mismatch / Checkout Conflict (serialize them).
- Connect cart/checkout **writes** need a shopper session (guest/community user); admin `sf api request rest` only works for **reads** (products/search/pricing) + management (reindex).
- **Shopper catalog 403/404** (guest AND buyer) is usually one of three, in order: (1) entitlement policy has no `CommerceEntitlementProduct` rows → 404; (2) buyer missing the **D2C Commerce Shopper PSL** → 403; (3) buyer missing the **`CommerceUser`** permset → 403. Object Read alone is NOT enough. Re-login after fixing.
- Custom LWR homepage components: place via the `DigitalExperienceBundle` home `content.json` (section `id` == `sectionConfig.UUID`); **clear `mobile.json`/`tablet.json` variant children** or deploy fails on stale component IDs; head-markup font `<link>` rejects `preconnect`/`crossorigin`.
- `Coupon.Status` defaults **Inactive**; a `PromotionSegment` is store-XOR-buyergroup; DML promotions may still need a Commerce-UI activation to actually apply.
- **Brand-aware chat (§18):** scope reco to the site by stamping `MessagingSession.Brand__c`; a **before-insert trigger deriving brand from `MessagingChannel.DeveloperName`** is the robust no-UI transport. The prechat-param route is **publish-gated** (a NEW param is dropped at runtime until the Embedded Service Deployment is Published — Setup-UI-only). **Agent linked vars resolve at session START** (mid-session field set is ignored). `sf agent publish` ≠ activate (must `sf agent activate --version N`). Agent's running user needs FLS on the MessagingSession field. Linked-var/action-input descriptions ≤255 chars.
